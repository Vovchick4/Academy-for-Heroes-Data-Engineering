# 📘 УРОК: Оптимізація SQL, Індекси та Партиціювання

---

# 🧠 1. Чому SQL може бути повільним

Основні причини:

* повний скан таблиці (**Full Table Scan**)
* відсутність індексів
* неправильні JOIN
* великі таблиці (мільйони+ рядків)
* погані WHERE умови

---

# ⚡ 2. Оптимізація запитів (Query Optimization)

## ❌ Погано

```sql
SELECT * FROM orders
WHERE YEAR(created_at) = 2024;
```

👉 проблема: функція `YEAR()` ламає індекс

---

## ✅ Добре

```sql
SELECT * FROM orders
WHERE created_at >= '2024-01-01'
AND created_at < '2025-01-01';
```

👉 тут індекс **може використовуватись**

---

## ❌ Погано

```sql
SELECT * FROM users
WHERE name LIKE '%ivan%';
```

👉 індекс не працює (через `%` на початку)

---

## ✅ Краще

```sql
WHERE name LIKE 'ivan%'
```

---

## ❌ SELECT *

```sql
SELECT * FROM orders;
```

👉 тягне зайві дані

---

## ✅ Краще

```sql
SELECT id, amount FROM orders;
```

# 📊 3. Індекси (Indexes)

Індекс — це **структура даних**, яка дозволяє швидко знаходити рядки.

👉 як “зміст книги”

---

## 🔹 Без індексу

SQL читає:

```
рядок 1 → рядок 2 → рядок 3 → ...
```

👉 повільно

---

## 🔹 З індексом

SQL робить:

```
пошук → дерево → результат
```

👉 дуже швидко

---

## 🔹 Створення індексу

```sql
CREATE INDEX idx_user_id ON orders(user_id);
```

---

## 🔹 Приклад

```sql
SELECT * FROM orders
WHERE user_id = 10;
```

👉 з індексом → швидко
👉 без → full scan

---

## 🔹 Composite Index (складений індекс)

```sql
CREATE INDEX idx_user_date 
ON orders(user_id, created_at);
```

---

### ⚠️ Важливо (правило зліва направо)

```sql
(user_id, created_at)
```

Працює для:

```sql
WHERE user_id = 1 ✅
WHERE user_id = 1 AND created_at > ... ✅
```

Не працює для:

```sql
WHERE created_at > ... ❌
```

---

## 🔹 Унікальний індекс

```sql
CREATE UNIQUE INDEX idx_email ON users(email);
```

---

## 🔹 Коли НЕ треба індекс

* маленька таблиця
* колонка з малою унікальністю (true/false)
* часто оновлюється (INSERT/UPDATE дорожчі)

---

# 🔍 4. EXPLAIN (найважливіше)

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 10;
```

Дивишся:

* `type` → ALL (погано), index, range (добре)
* `key` → який індекс використовується
* `rows` → скільки рядків читає

---

# 🔗 5. JOIN оптимізація

## ❌ Погано

```sql
SELECT *
FROM orders o
JOIN users u ON o.user_id = u.id;
```

без індексу на `user_id`

---

## ✅ Добре

```sql
CREATE INDEX idx_user_id ON orders(user_id);
```

👉 JOIN стає швидким

# 🧱 6. Партиціювання (Partitioning)

Партиціювання — це **розбиття великої таблиці на частини**.

👉 але виглядає як одна таблиця

---

## 🔹 Навіщо

* пришвидшити запити
* працювати з великими даними (мільйони+)
* легше видаляти старі дані

---

## 🔹 Приклад (по даті)

```sql
PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025)
);
```

---

## 🔹 Як це працює

Запит:

```sql
SELECT * FROM orders
WHERE created_at >= '2024-01-01';
```

👉 SQL читає **тільки partition 2024**

👉 не сканує всю таблицю

---

## 🔹 Типи partition

### 1. RANGE

```sql
по даті / числу
```

---

### 2. LIST

```sql
по значенням (status = 'new', 'done')
```

---

### 3. HASH

```sql
розподіл рівномірно
```

---

# ⚖️ 7. Partition vs Index

|           | Index         | Partition                  |
| --------- | ------------- | -------------------------- |
| Що робить | швидкий пошук | менше даних для сканування |
| Коли      | часті WHERE   | дуже великі таблиці        |
| Розмір    | не змінює     | ділить таблицю             |

---

# 🔥 8. Реальний кейс (як на співбесіді)

### ❌ Повільний запит

```sql
SELECT *
FROM orders
WHERE user_id = 10
AND created_at >= '2024-01-01';
```

---

## ✅ Оптимізація

1. Індекс:

```sql
CREATE INDEX idx_user_date 
ON orders(user_id, created_at);
```

2. Якщо таблиця велика:

👉 додаємо partition по даті

---

# 🧠 9. Best Practices

* завжди індексуй:

  * `WHERE`
  * `JOIN`
  * `ORDER BY`
* не використовуй функції в WHERE
* використовуй `EXPLAIN`
* не роби `SELECT *`
* використовуй composite indexes
* думай про **об’єм даних**

---
