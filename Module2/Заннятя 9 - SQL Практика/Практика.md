## 1. Основи: DDL, DML та Типи даних
Для інженера даних важливо не тільки створити таблицю, а й правильно обрати типи даних для економії пам'яті та швидкості обробки.

### Приклад (Створення схеми "Зірка"):
```sql
-- DDL: Створення таблиці фактів
CREATE TABLE sales (
    sale_id SERIAL PRIMARY KEY,
    product_id INT NOT NULL,
    sale_date DATE DEFAULT CURRENT_DATE,
    amount DECIMAL(12, 2), -- Точність для фінансів
    metadata JSONB -- Для неструктурованих даних
);

-- DML: Масове вставлення
INSERT INTO sales (product_id, amount)
VALUES (101, 250.50), (102, 89.99);
```

## 2. JOIN-запити та CTE
У Data Engineering ми часто використовуємо **CTE (Common Table Expressions)** замість вкладених підзапитів для кращої читаємості коду (Readable Code).

### Завдання:
Знайти всіх користувачів, які не зробили жодного замовлення за останній місяць.

### Рішення через CTE:
```sql
WITH recent_orders AS (
    SELECT DISTINCT user_id 
    FROM orders 
    WHERE order_date > CURRENT_DATE - INTERVAL '30 days'
)
SELECT u.user_name, u.email
FROM users u
LEFT JOIN recent_orders ro ON u.id = ro.user_id
WHERE ro.user_id IS NULL;
```

## 3. Віконні функції (Window Functions)
Це "хліб" дата-інженера. Вони дозволяють робити обчислення поверх набору рядків, не групуючи їх в один.



### Приклад: Ранжування продажів
```sql
SELECT 
    product_id,
    sale_date,
    amount,
    SUM(amount) OVER (PARTITION BY product_id ORDER BY sale_date) as running_total,
    RANK() OVER (ORDER BY amount DESC) as rank_by_amount
FROM sales;
```
* `SUM(...) OVER ...` — створює кумулятивну суму (Running Total).
* `RANK()` — присвоює місце товару в рейтингу продажів.

## 4. Оптимізація: Індексування та Партиціювання
Коли таблиця має мільярди рядків, звичайний пошук зупиняється.

* **Індекси (B-Tree, GIN):** Пришвидшують пошук, але сповільнюють запис.
* **Партиціювання:** Розбиття великої таблиці на фізичні частини (наприклад, за датами).

### Приклад партиціювання за часом:
```sql
CREATE TABLE sensor_data (
    sensor_id INT,
    reading_time TIMESTAMP,
    value FLOAT
) PARTITION BY RANGE (reading_time);

-- Створення партиції для 2026 року
CREATE TABLE sensor_data_2026_01 PARTITION OF sensor_data
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
```

## 5. Складні SQL-завдання для практики

### Завдання 1: Пошук "Gaps and Islands" (Прогалини та острови)
**Контекст:** Маємо лог активності сервера. Треба знайти періоди (початок і кінець), коли сервер був онлайн без перерв.

```sql
WITH grp_calculation AS (
    SELECT 
        event_time,
        event_time - (ROW_NUMBER() OVER (ORDER BY event_time) * INTERVAL '1 minute') as grp
    FROM server_logs
    WHERE status = 'online'
)
SELECT MIN(event_time), MAX(event_time)
FROM grp_calculation
GROUP BY grp;
```

### Завдання 2: Очищення дублікатів (Data Cleaning)
**Контекст:** У ETL-процес потрапили дублікати. Треба видалити повні копії, залишивши лише найсвіжіший запис за `id`.

```sql
DELETE FROM users
WHERE row_id IN (
    SELECT row_id FROM (
        SELECT row_id,
               ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY updated_at DESC) as rn
        FROM users
    ) t WHERE t.rn > 1
);
```

## Поради щодо оптимізації запитів
1.  **Уникайте `SELECT *`**: Витягуйте лише ті колонки, які потрібні для Pipeline.
2.  **Sargable predicates**: Замість `WHERE YEAR(date) = 2024` використовуйте `WHERE date >= '2024-01-01' AND date < '2025-01-01'`. Це дозволяє використовувати індекси.
3.  **Explain Analyze**: Завжди перевіряйте план виконання запиту (`EXPLAIN ANALYZE your_query`), щоб побачити "вузькі місця" (наприклад, Seq Scan на великих таблицях).
