# Частина 1 — Агрегація даних

(30–40 хв)

У реальному Data Engineering **90% SQL запитів — це агрегування**.

Агрегація — це коли ми **об'єднуємо багато рядків у один результат**.

Наприклад:

| order_id | user | price |
| -------- | ---- | ----- |
| 1        | Ivan | 100   |
| 2        | Ivan | 200   |
| 3        | Anna | 300   |

Можемо порахувати:

* загальну суму
* середню ціну
* кількість замовлень

---

## Основні агрегатні функції

| Функція | Опис                 |
| ------- | -------------------- |
| COUNT() | кількість рядків     |
| SUM()   | сума                 |
| AVG()   | середнє              |
| MAX()   | максимальне значення |
| MIN()   | мінімальне значення  |

---

## Приклад таблиці

```sql
CREATE TABLE orders (
    id INT,
    user_name VARCHAR(50),
    amount INT
);
```

Дані:

```sql
INSERT INTO orders VALUES
(1,'Ivan',100),
(2,'Ivan',200),
(3,'Anna',300),
(4,'Anna',150),
(5,'Oleh',400);
```

---

## COUNT()

Порахувати кількість замовлень

```sql
SELECT COUNT(*) 
FROM orders;
```

Результат

```
5
```

---

## SUM()

Сума всіх продажів

```sql
SELECT SUM(amount)
FROM orders;
```

---

## AVG()

Середній чек

```sql
SELECT AVG(amount)
FROM orders;
```

---

# Частина 2 — GROUP BY

(40 хв)

`GROUP BY` дозволяє **групувати рядки по колонці**.

Наприклад:

Порахувати **суму продажів для кожного користувача**

```sql
SELECT user_name, SUM(amount)
FROM orders
GROUP BY user_name;
```

Результат:

| user_name | sum |
| --------- | --- |
| Ivan      | 300 |
| Anna      | 450 |
| Oleh      | 400 |

---

## Кількість замовлень на користувача

```sql
SELECT user_name, COUNT(*)
FROM orders
GROUP BY user_name;
```

---

## Середній чек

```sql
SELECT user_name, AVG(amount)
FROM orders
GROUP BY user_name;
```

---

# Частина 3 — HAVING

(20 хв)

`HAVING` використовується **для фільтрації GROUP BY**.

Важливо:

```
WHERE → фільтрує рядки
HAVING → фільтрує групи
```

---

## Приклад

Користувачі з сумою замовлень більше 400

```sql
SELECT user_name, SUM(amount)
FROM orders
GROUP BY user_name
HAVING SUM(amount) > 400;
```

Результат:

| user_name | sum |
| --------- | --- |
| Anna      | 450 |

---

# Частина 4 — JOIN (З'єднання таблиць)

У реальних базах даних **інформація майже ніколи не зберігається в одній таблиці**.

Замість цього дані **нормалізуються** і розбиваються на окремі таблиці.

Наприклад:

* користувачі
* замовлення
* продукти
* платежі

Тому для отримання повної інформації потрібно **об'єднувати таблиці**.

Це робиться за допомогою **JOIN**.

---

# Що таке JOIN

**JOIN** — це операція SQL, яка дозволяє **об'єднати рядки з двох або більше таблиць на основі пов'язаних колонок**.

Найчастіше це:

```
PRIMARY KEY  →  FOREIGN KEY
```

Наприклад:

```
users.id = orders.user_id
```

Тобто:

* `users.id` — унікальний ідентифікатор користувача
* `orders.user_id` — посилання на користувача

---

# Приклад таблиць

### users

| id | name |
| -- | ---- |
| 1  | Ivan |
| 2  | Anna |
| 3  | Oleh |

---

### orders

| id | user_id | amount |
| -- | ------- | ------ |
| 1  | 1       | 100    |
| 2  | 1       | 200    |
| 3  | 2       | 300    |

---

# Як працює JOIN

SQL фактично **порівнює значення колонок** і об'єднує рядки.

Алгоритм простими словами:

```
для кожного рядка таблиці users
    знайти рядки orders де users.id = orders.user_id
    об'єднати їх
```

---

# Типи JOIN

У SQL існує кілька типів JOIN:

| JOIN       | Опис                       |
| ---------- | -------------------------- |
| INNER JOIN | тільки співпадаючі рядки   |
| LEFT JOIN  | всі рядки з лівої таблиці  |
| RIGHT JOIN | всі рядки з правої таблиці |
| FULL JOIN  | всі рядки з обох таблиць   |
| CROSS JOIN | декартовий добуток         |

---

# 1 INNER JOIN

`INNER JOIN` повертає **тільки рядки які мають співпадіння у двох таблицях**.

Якщо зв'язку немає — рядок не потрапляє у результат.

---

## SQL

```sql
SELECT users.name, orders.amount
FROM users
INNER JOIN orders
ON users.id = orders.user_id;
```

---

## Результат

| name | amount |
| ---- | ------ |
| Ivan | 100    |
| Ivan | 200    |
| Anna | 300    |

---

## Чому Oleh не з'явився?

Тому що:

```
users.id = 3
```

але

```
orders.user_id = 3
```

немає.

Тобто **немає співпадіння**.

---

# 2 LEFT JOIN

`LEFT JOIN` повертає:

```
всі рядки з лівої таблиці
+
співпадаючі рядки з правої
```

Якщо співпадіння немає — SQL підставляє `NULL`.

---

## SQL

```sql
SELECT users.name, orders.amount
FROM users
LEFT JOIN orders
ON users.id = orders.user_id;
```

---

## Результат

| name | amount |
| ---- | ------ |
| Ivan | 100    |
| Ivan | 200    |
| Anna | 300    |
| Oleh | NULL   |

---

## Чому NULL?

Тому що:

```
Oleh не має orders
```

---

# 3 RIGHT JOIN

`RIGHT JOIN` — це дзеркальна версія `LEFT JOIN`.

Повертає:

```
всі рядки з правої таблиці
+
співпадіння з лівої
```

---

## SQL

```sql
SELECT users.name, orders.amount
FROM users
RIGHT JOIN orders
ON users.id = orders.user_id;
```

---

## Результат

| name | amount |
| ---- | ------ |
| Ivan | 100    |
| Ivan | 200    |
| Anna | 300    |

У нашому прикладі результат той самий, тому що **всі orders мають user_id**.

---

# 4 FULL JOIN

`FULL JOIN` повертає:

```
всі рядки з обох таблиць
```

Навіть якщо немає співпадіння.

---

⚠️ Важливо

У **MySQL FULL JOIN не підтримується напряму**.

Його можна емулювати через:

```sql
LEFT JOIN
UNION
RIGHT JOIN
```

---

# 5 CROSS JOIN

`CROSS JOIN` створює **декартовий добуток**.

Тобто **кожен рядок першої таблиці з'єднується з кожним рядком другої**.

---

### Приклад

users

| id |
| -- |
| 1  |
| 2  |

products

| id |
| -- |
| A  |
| B  |

---

Результат

| user | product |
| ---- | ------- |
| 1    | A       |
| 1    | B       |
| 2    | A       |
| 2    | B       |

---

## SQL

```sql
SELECT *
FROM users
CROSS JOIN products;
```

---

# JOIN + GROUP BY

Дуже часто JOIN використовується разом з **агрегацією**.

Наприклад:

**загальна сума замовлень кожного користувача**

---

```sql
SELECT users.name, SUM(orders.amount)
FROM users
JOIN orders
ON users.id = orders.user_id
GROUP BY users.name;
```

---

## Результат

| name | total |
| ---- | ----- |
| Ivan | 300   |
| Anna | 300   |

---

# Візуальне пояснення JOIN

Уявіть **Venn діаграми**.

### INNER JOIN

```
users ∩ orders
```

тільки перетин.

---

### LEFT JOIN

```
users + intersection
```

все з users.

---

### RIGHT JOIN

```
orders + intersection
```

все з orders.

---

# Коли використовують JOIN у реальних проектах

JOIN — це **одна з найчастіших операцій у SQL**.

Приклади:

### E-commerce

```
users + orders
```

щоб отримати покупки користувача.

---

### Analytics

```
orders + products
```

щоб порахувати продажі продуктів.

---

### Fintech

```
transactions + users
```

щоб знайти історію платежів.

---

# Типові помилки JOIN

### 1 Неправильна колонка

❌

```sql
ON users.id = orders.id
```

Правильно

```
users.id = orders.user_id
```

---

### 2 Забутий ON

❌

```sql
SELECT *
FROM users
JOIN orders;
```

Це створить **CROSS JOIN**.

---

### 3 Неправильна агрегація

❌

```sql
SELECT users.name, orders.amount
FROM users
JOIN orders
GROUP BY users.name
```

# Частина 5 — Пошук дублікатів

(20 хв)

Дуже часта задача у Data Engineering.

Наприклад таблиця:

| email                             |
| --------------------------------- |
| [a@gmail.com](mailto:a@gmail.com) |
| [b@gmail.com](mailto:b@gmail.com) |
| [a@gmail.com](mailto:a@gmail.com) |

---

## Знайти дублікати

```sql
SELECT email, COUNT(*)
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

Результат:

| email                             | count |
| --------------------------------- | ----- |
| [a@gmail.com](mailto:a@gmail.com) | 2     |

# Частина 7 — Stored Procedures

(20 хв)

Stored Procedure — це **SQL код який зберігається в базі**.

Використовується для:

* складних запитів
* ETL процесів
* автоматизації

---

## Приклад

```sql
CREATE PROCEDURE get_orders()
BEGIN
    SELECT * FROM orders;
END;
```

Виклик

```sql
CALL get_orders();
```

---

## Procedure з параметром

```sql
CREATE PROCEDURE get_user_orders(IN uid INT)
BEGIN
    SELECT * 
    FROM orders
    WHERE user_id = uid;
END;
```

Виклик

```sql
CALL get_user_orders(1);
```
