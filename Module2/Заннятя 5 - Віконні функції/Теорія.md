**Віконні функції (Window Functions) в SQL** — це функції, які дозволяють виконувати обчислення **над набором рядків, пов’язаних з поточним рядком**, не згортаючи їх у один результат (як це робить `GROUP BY`).

Тобто вони **рахують агрегати, але залишають усі рядки**.

---

# 1. Синтаксис

```sql
FUNCTION_NAME(column)
OVER (
    PARTITION BY column
    ORDER BY column
)
```

**Частини:**

* `FUNCTION_NAME` — функція (`SUM`, `AVG`, `ROW_NUMBER` тощо)
* `OVER()` — визначає **вікно**
* `PARTITION BY` — розбиває дані на групи
* `ORDER BY` — порядок обчислення

---

# 2. Приклад таблиці

### orders

| id | user_id | amount |
| -- | ------- | ------ |
| 1  | 1       | 100    |
| 2  | 1       | 200    |
| 3  | 2       | 150    |
| 4  | 2       | 300    |
| 5  | 3       | 50     |

# 3. SUM як віконна функція

```sql
SELECT 
    id,
    user_id,
    amount,
    SUM(amount) OVER(PARTITION BY user_id) AS total_user
FROM orders;
```

Результат

| id | user_id | amount | total_user |
| -- | ------- | ------ | ---------- |
| 1  | 1       | 100    | 300        |
| 2  | 1       | 200    | 300        |
| 3  | 2       | 150    | 450        |
| 4  | 2       | 300    | 450        |
| 5  | 3       | 50     | 50         |

Тут:

```
PARTITION BY user_id
```

означає:

**рахувати суму для кожного користувача**

Ось **коротша теорія**, яку можна додати до цього прикладу.

---

### Що робить `SUM() OVER()`

```sql
SELECT 
    id,
    user_id,
    amount,
    SUM(amount) OVER(PARTITION BY user_id) AS total_user
FROM orders;
```

`SUM()` — агрегатна функція.
`OVER()` перетворює її у **віконну функцію**.

Це означає, що:

* функція рахує значення **для групи рядків**
* **але не об’єднує рядки**, як `GROUP BY`

Тому кожен рядок залишається в результаті.

---

### `PARTITION BY`

```
PARTITION BY user_id
```

Розбиває таблицю на **групи (partition)** за `user_id`.

Для кожної групи SQL:

1. бере всі рядки з однаковим `user_id`
2. рахує `SUM(amount)`
3. записує результат у **кожен рядок цієї групи**

---

### Чим відрізняється від `GROUP BY`

`GROUP BY`

```sql
SELECT user_id, SUM(amount)
FROM orders
GROUP BY user_id;
```

→ повертає **один рядок на групу**

`SUM() OVER(PARTITION BY ...)`

→ залишає **усі рядки**, але додає обчислене значення.

---

### Якщо `PARTITION BY` не використовувати

```sql
SUM(amount) OVER()
```

SQL порахує суму **по всій таблиці** і запише її в кожний рядок.

---

# 4. `ROW_NUMBER()`

Нумерує рядки.

```sql
SELECT 
    id,
    user_id,
    amount,
    ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY amount DESC) AS rank
FROM orders;
```

Результат

| user_id | amount | rank |
| ------- | ------ | ---- |
| 1       | 200    | 1    |
| 1       | 100    | 2    |
| 2       | 300    | 1    |
| 2       | 150    | 2    |

### Теорія

`ROW_NUMBER()` — це **віконна функція**, яка присвоює **послідовний номер кожному рядку** у межах визначеного вікна.

У цьому запиті відбувається таке:

1. `PARTITION BY user_id`
   Таблиця розбивається на **групи за користувачами**.

   Тобто SQL окремо працює з рядками:

   * user 1
   * user 2
   * user 3 (якщо є)

2. `ORDER BY amount DESC`
   У кожній групі рядки **сортуються за сумою замовлення від найбільшої до найменшої**.

3. `ROW_NUMBER()`
   Після сортування SQL **присвоює порядковий номер кожному рядку**:

   ```
   1, 2, 3, 4 ...
   ```

   Нумерація починається **знову для кожної групи**.

### Що відбувається на прикладі

Для `user_id = 1`:

| amount |
| ------ |
| 200    |
| 100    |

Після сортування:

| amount | rank |
| ------ | ---- |
| 200    | 1    |
| 100    | 2    |

Для `user_id = 2`:

| amount |
| ------ |
| 300    |
| 150    |

Після сортування:

| amount | rank |
| ------ | ---- |
| 300    | 1    |
| 150    | 2    |

### Важливо

`ROW_NUMBER()` **завжди присвоює унікальний номер** кожному рядку, навіть якщо значення однакові.

---

# 5. `RANK()`

`RANK()` — це віконна функція, яка **присвоює ранг рядкам на основі сортування**.

Якщо значення **однакові**, вони отримують **однаковий ранг**.

```sql
SELECT 
    amount,
    RANK() OVER(ORDER BY amount DESC) AS rank
FROM orders;
```

### Як працює

1. `ORDER BY amount DESC`
   SQL **сортує всі рядки за сумою від більшої до меншої**.

2. `RANK()`
   Після сортування кожному рядку присвоюється **позиція (ранг)**.

---

### Приклад

| amount |
| ------ |
| 300    |
| 200    |
| 200    |
| 100    |

Результат:

| amount | rank |
| ------ | ---- |
| 300    | 1    |
| 200    | 2    |
| 200    | 2    |
| 100    | 4    |

---

### Важлива особливість

Якщо значення **повторюються**, вони отримують **однаковий ранг**, але наступний ранг **пропускається**.

У прикладі:

```
1
2
2
4
```

Ранг **3 пропущений**, тому що два рядки мають ранг **2**.

---

### Де використовують

`RANK()` часто застосовують для:

* **рейтингу продажів**
* **лідербордів**
* **аналізу топ-продуктів**
* **ранжування користувачів або замовлень**.


---

# 6. `DENSE_RANK()`

`DENSE_RANK()` — це віконна функція, яка **присвоює ранг рядкам на основі сортування**, подібно до `RANK()`.

Головна різниця:
**ранги не пропускаються**, навіть якщо є однакові значення.

```sql
SELECT 
    amount,
    DENSE_RANK() OVER(ORDER BY amount DESC) AS rank
FROM orders;
```

### Як працює

1. `ORDER BY amount DESC`
   SQL спочатку **сортує рядки від найбільшого значення до найменшого**.

2. `DENSE_RANK()`
   Після сортування кожному рядку присвоюється **ранг**.

Якщо значення **однакові**, вони отримують **однаковий ранг**, але наступний ранг **не пропускається**.

---

### Приклад

| amount |
| ------ |
| 300    |
| 200    |
| 200    |
| 100    |

Результат:

| amount | rank |
| ------ | ---- |
| 300    | 1    |
| 200    | 2    |
| 200    | 2    |
| 100    | 3    |

---

### Порівняння з `RANK()`

| amount | RANK | DENSE_RANK |
| ------ | ---- | ---------- |
| 300    | 1    | 1          |
| 200    | 2    | 2          |
| 200    | 2    | 2          |
| 100    | 4    | 3          |

* `RANK()` → пропускає ранг
* `DENSE_RANK()` → **ранги йдуть підряд**

---

### Де використовують

`DENSE_RANK()` часто застосовують для:

* рейтингів
* топ-списків
* аналізу продажів
* визначення позиції елемента у відсортованому наборі даних.

---

# 7. Running total (накопичувальна сума)

```sql
SELECT 
    id,
    amount,
    SUM(amount) OVER(ORDER BY id) AS running_total
FROM orders;
```

Результат

| id | amount | running_total |
| -- | ------ | ------------- |
| 1  | 100    | 100           |
| 2  | 200    | 300           |
| 3  | 150    | 450           |
| 4  | 300    | 750           |

### Теорія

**Running total** (накопичувальна сума) — це сума значень **від першого рядка до поточного**.

У цьому запиті відбувається таке:

1. `ORDER BY id`
   SQL спочатку **сортує рядки за `id`**.

2. `SUM(amount) OVER(...)`
   Функція `SUM()` рахує **суму значень у вікні**, яке включає:

   * всі попередні рядки
   * поточний рядок.

Тобто для кожного рядка SQL **додає суму попередніх значень**.

---

### Як формується результат

| id | amount | обчислення            | running_total |
| -- | ------ | --------------------- | ------------- |
| 1  | 100    | 100                   | 100           |
| 2  | 200    | 100 + 200             | 300           |
| 3  | 150    | 100 + 200 + 150       | 450           |
| 4  | 300    | 100 + 200 + 150 + 300 | 750           |

---

### Де використовують

Накопичувальні суми часто використовують для:

* **фінансових звітів**
* **аналізу продажів у часі**
* **балансу рахунків**
* **аналітики часових рядів (time series)**.

Це один з найпоширеніших типів аналітичних запитів у SQL.

---

# 8. `LAG()`

`LAG()` — це віконна функція, яка дозволяє **отримати значення з попереднього рядка**.

```sql
SELECT 
    id,
    amount,
    LAG(amount) OVER(ORDER BY id) AS previous_amount
FROM orders;
```

### Теорія

Функція `LAG()` використовується, коли потрібно **порівняти поточний рядок з попереднім**.

У цьому запиті відбувається таке:

1. `ORDER BY id`
   SQL спочатку **сортує рядки за `id`**.

2. `LAG(amount)`
   Для кожного рядка SQL бере **значення `amount` з попереднього рядка**.

---

### Приклад

| id | amount |
| -- | ------ |
| 1  | 100    |
| 2  | 200    |
| 3  | 150    |
| 4  | 300    |

Результат:

| id | amount | previous_amount |
| -- | ------ | --------------- |
| 1  | 100    | NULL            |
| 2  | 200    | 100             |
| 3  | 150    | 200             |
| 4  | 300    | 150             |

Перший рядок має `NULL`, тому що **попереднього рядка не існує**.

---

### Де використовують

`LAG()` часто використовують для:

* **порівняння значень між рядками**
* **аналізу змін у часі**
* **обчислення різниці між періодами**
* **фінансової та часової аналітики**.

---

# 9. `LEAD()`

`LEAD()` — це віконна функція, яка дозволяє **отримати значення з наступного рядка**.

```sql
SELECT 
    id,
    amount,
    LEAD(amount) OVER(ORDER BY id) AS next_amount
FROM orders;
```

### Теорія

Функція `LEAD()` працює подібно до `LAG()`, але дивиться **не назад, а вперед**.

У цьому запиті:

1. `ORDER BY id`
   SQL **сортує рядки за `id`**.

2. `LEAD(amount)`
   Для кожного рядка SQL бере **значення `amount` з наступного рядка**.

---

### Приклад

| id | amount |
| -- | ------ |
| 1  | 100    |
| 2  | 200    |
| 3  | 150    |
| 4  | 300    |

Результат:

| id | amount | next_amount |
| -- | ------ | ----------- |
| 1  | 100    | 200         |
| 2  | 200    | 150         |
| 3  | 150    | 300         |
| 4  | 300    | NULL        |

Останній рядок має `NULL`, тому що **наступного рядка не існує**.

---

### Де використовують

`LEAD()` часто застосовують для:

* **порівняння значень між рядками**
* **аналізу змін між періодами**
* **обчислення різниці між поточним і наступним значенням**
* **аналізу часових даних**.

---

# 10. Різниця з GROUP BY

### GROUP BY

```sql
SELECT user_id, SUM(amount)
FROM orders
GROUP BY user_id
```

Результат

| user_id | sum |
| ------- | --- |
| 1       | 300 |
| 2       | 450 |

**Рядки об'єднуються**

---

### Window function

```sql
SUM(amount) OVER(PARTITION BY user_id)
```

Рядки **не зникають**

# 11. Frame (рамка вікна)

Frame (рамка) визначає **які саме рядки всередині вікна будуть використовуватись для обчислення функції**.

За допомогою рамки можна обмежити кількість рядків, які бере SQL для розрахунку.

```sql
SUM(amount) OVER(
    ORDER BY id
    ROWS BETWEEN 1 PRECEDING AND CURRENT ROW
)
```

### Теорія

1. `ORDER BY id`
   Спочатку SQL **сортує рядки**.

2. `ROWS BETWEEN ... AND ...`
   Це визначає **межі вікна (frame)** — тобто які рядки враховуються при обчисленні.

У цьому прикладі:

```
ROWS BETWEEN 1 PRECEDING AND CURRENT ROW
```

означає:

* **1 PRECEDING** → один попередній рядок
* **CURRENT ROW** → поточний рядок

Тобто функція `SUM()` рахує суму **поточного рядка + попереднього**.

---

### Приклад

| id | amount |
| -- | ------ |
| 1  | 100    |
| 2  | 200    |
| 3  | 150    |
| 4  | 300    |

Результат:

| id | amount | sum |
| -- | ------ | --- |
| 1  | 100    | 100 |
| 2  | 200    | 300 |
| 3  | 150    | 350 |
| 4  | 300    | 450 |

Обчислення:

| id | формула   |
| -- | --------- |
| 1  | 100       |
| 2  | 100 + 200 |
| 3  | 200 + 150 |
| 4  | 150 + 300 |

---

### Навіщо це використовують

Frame дозволяє робити:

* **ковзні суми (moving sum)**
* **ковзні середні (moving average)**
* **аналіз часових рядів**
* **фінансову аналітику**.

Тобто замість використання **всіх попередніх рядків**, можна обмежити обчислення **лише кількома сусідніми рядками**.

---

✅ **Класичне інтерв'ю питання**

Знайти **найдорожче замовлення кожного користувача**

```sql
SELECT *
FROM (
    SELECT *,
           ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY amount DESC) AS rn
    FROM orders
) t
WHERE rn = 1;
```
