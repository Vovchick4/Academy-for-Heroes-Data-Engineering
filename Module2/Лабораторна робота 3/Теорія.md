---

# 🔎 Що ми робимо в цій лабораторній?

У попередніх лабораторних:

* ми працювали з **OLTP-базою**
* аналізували структуру
* будували ERD
* дивилися як працюють транзакційні таблиці

❗ Але OLTP — це не для аналітики.

Тепер ми:

👉 будуємо **Data Warehouse (аналітичну модель)**
👉 перетворюємо нормалізовану структуру в **денормалізовану Star Schema**
👉 готуємо дані для BI / Power BI / аналітики

# 🧠 Крок 1. Усвідомлення бізнес-запиту

# 🧱 Крок 2. Визначаємо Grain (рівень деталізації)

🔴 Найважливіший момент у DWH.

**Grain фактової таблиці:**

> Один рядок = один рядок замовлення (SalesOrderDetail)

Тобто:

* не один заказ
* не один клієнт
* а саме один проданий товар в замовленні

## Чому саме так?

Бо фактова таблиця:

* росте постійно
* містить числа

# 📘 Крок 4. Визначаємо Dimension таблиці

## 1️⃣ DimDate

Чому потрібна окрема таблиця дат?

Бо в аналітиці ми хочемо:

* групувати по роках
* по кварталах
* по місяцях
* робити time intelligence

У OLTP такого немає.

---

## 2️⃣ DimProduct

В OLTP:

```
Product → SubCategory → Category
```

У DWH:

```
DimProduct
  - ProductName
  - SubCategory
  - Category
```

Ми денормалізуємо.

Менше JOIN = швидше аналітика.

---

## 3️⃣ DimCustomer

В OLTP клієнт розбитий на:

* Customer
* Person
* Address
* CountryRegion

У DWH:

Все в одну таблицю.

# 🔄 Крок 7. ETL – що відбувається реально?

ETL = Extract → Transform → Load

## 🔹 Extract

Беремо дані з OLTP (AdventureWorks)

## 🔹 Transform

* об'єднуємо таблиці
* денормалізуємо
* створюємо surrogate keys

## 🔹 Load

Вставляємо в DW

```sql
INSERT INTO DimProduct (ProductID, ProductName, SubCategory, Category)
SELECT 
    p.ProductID,
    p.Name,
    sc.Name,
    c.Name
FROM AdventureWorks2025.Production.Product p
LEFT JOIN AdventureWorks2025.Production.ProductSubcategory sc
    ON p.ProductSubcategoryID = sc.ProductSubcategoryID
LEFT JOIN AdventureWorks2025.Production.ProductCategory c
    ON sc.ProductCategoryID = c.ProductCategoryID;
```

AdventureWorks2025 - Ваша База данних яку робили на ЛАБ2

📌 Що відбувається:
беремо дані з OLTP
денормалізуємо
завантажуємо в DWH

```sql
INSERT INTO DimCustomer (CustomerID, FirstName, LastName)
SELECT 
    c.CustomerID,
    p.FirstName,
    p.LastName
FROM AdventureWorks2025.Sales.Customer c
JOIN AdventureWorks2025.Person.Person p
    ON c.PersonID = p.BusinessEntityID;
```

```sqL
INSERT INTO DimDate
SELECT DISTINCT
    CONVERT(INT, FORMAT(OrderDate, 'yyyyMMdd')) AS DateKey,
    OrderDate,
    YEAR(OrderDate),
    DATEPART(QUARTER, OrderDate),
    MONTH(OrderDate),
    DATENAME(MONTH, OrderDate)
FROM AdventureWorks2025.Sales.SalesOrderHeader;
```

# 🧠 ВАЖЛИВО: Surrogate Key

Чому в DimProduct є:

```sql
ProductKey INT IDENTITY
```

А не просто ProductID?

Бо:

* ProductID = бізнес ключ
* ProductKey = технічний ключ DWH
* дозволяє робити SCD Type 2
* не залежить від OLTP

# 🔥 Крок 8. Завантаження Fact

Коли ми вставляємо в Fact:

```sql
CREATE TABLE FactSales (
    SalesKey INT PRIMARY KEY IDENTITY(1,1),
    DateKey INT,
    ProductKey INT,
    CustomerKey INT,
    OrderQuantity INT,
    SalesAmount DECIMAL(18,2),

    FOREIGN KEY (DateKey) REFERENCES DimDate(DateKey),
    FOREIGN KEY (ProductKey) REFERENCES DimProduct(ProductKey),
    FOREIGN KEY (CustomerKey) REFERENCES DimCustomer(CustomerKey)
);
```

```sql
INSERT INTO FactSales WITH (TABLOCK)
(DateKey, ProductKey, CustomerKey, OrderQuantity, SalesAmount)
SELECT
    YEAR(h.OrderDate) * 10000 +
    MONTH(h.OrderDate) * 100 +
    DAY(h.OrderDate),
    dp.ProductKey,
    dc.CustomerKey,
    s.OrderQty,
    s.LineTotal
FROM AdventureWorks2025.Sales.SalesOrderDetail s
JOIN AdventureWorks2025.Sales.SalesOrderHeader h 
    ON s.SalesOrderID = h.SalesOrderID
JOIN DimProduct dp 
    ON s.ProductID = dp.ProductID
JOIN DimCustomer dc 
    ON h.CustomerID = dc.CustomerID;
```

Ми:
* шукаємо surrogate key
* замінюємо бізнес ключ на технічний
* зв'язуємо з вимірами

🎯 Тепер структура готова

# 🧠 Головна ідея цієї роботи

OLTP-система створена для:

* зберігання транзакцій
* забезпечення цілісності даних
* швидкого виконання CRUD-операцій

Але вона:

* складна для аналітики
* містить багато JOIN
* має глибоку нормалізацію

Data Warehouse створюється для:

* аналітики
* агрегування
* швидких звітів
* BI-систем

---

# ⭐ Чому схема «Зірка» така важлива?

Star Schema:

* спрощує структуру
* зменшує кількість JOIN
* прискорює агрегації
* зрозуміла для аналітиків
* легко інтегрується з Power BI / Tableau

В центрі — Fact таблиця (числа).
Навколо — Dimension таблиці (контекст).

Це дозволяє швидко відповідати на питання бізнесу.

---

# 📊 Для чого це в реальному світі?

У реальних компаніях:

* OLTP = ERP / CRM / eCommerce система
* DWH = окрема аналітична база

Аналітики не працюють з продакшн-базою напряму.
Вони працюють з Data Warehouse.

Тому що:

* це безпечніше
* це швидше
* це стабільніше
* це масштабованіше

# 🔥 Що найважливіше ми зрозуміли

### 1️⃣ Grain — основа всього

Якщо неправильно визначити рівень деталізації — вся модель буде некоректна.

### 2️⃣ Fact ≠ Dimension

Fact — це цифри.
Dimension — це опис.

### 3️⃣ Surrogate Key — критично важливо

Технічні ключі дозволяють:

* реалізувати SCD
* не залежати від OLTP
* контролювати історичність

### 4️⃣ Денормалізація — це не помилка

В OLTP ми нормалізуємо.
В DWH — спеціально денормалізуємо.
